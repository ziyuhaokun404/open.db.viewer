## 2. MVP 定义

本 MVP 的目标不是做一个完整的数据库 IDE，而是做出一个可稳定使用的轻量数据库浏览工具，帮助用户完成以下最核心闭环：

1. 创建数据库连接
2. 成功连接数据库
3. 浏览数据库对象
4. 查看表结构
5. 查看表数据
6. 执行简单查询
7. 导出当前结果

只要上述闭环稳定成立，MVP 就具备上线验证价值。

## 3. MVP 目标

### 3.1 用户目标

用户可以在第一次使用时，以较低学习成本完成：

- 连接 MySQL / PostgreSQL / SQLite
- 找到目标数据库对象
- 查看一张表的结构
- 查看一张表的数据
- 执行一条简单查询
- 导出查询结果

### 3.2 产品目标

- 验证“轻量、易上手、聚焦浏览”的定位是否成立
- 验证用户是否愿意把产品作为日常数据库查看工具
- 为后续扩展多标签、查询历史、收藏对象等功能建立基础

### 3.3 成功标准

- 用户能够在 3 分钟内完成首次连接并进入浏览状态
- 用户能够在 3 次以内主要交互中找到目标表
- 用户能够无须额外教程完成查看结构、查看数据、执行简单查询
- 用户能稳定导出当前结果集

## 4. MVP 范围原则

### 4.1 必须纳入

- 直接支撑核心闭环的功能
- 高频操作必须依赖的基础能力
- 明显影响可用性与稳定性的底层支撑能力

### 4.2 可以延后

- 提升效率但不影响闭环成立的增强项
- 面向重度用户的高级能力
- 明显增加复杂度但对首版验证价值有限的功能

### 4.3 明确不做

- 与“快速连接、浏览、查看”无直接关系的扩展系统
- 团队协作、审批、安全治理、插件平台等企业能力
- 深度 SQL IDE 能力

## 5. MVP 功能分层

建议将 MVP 功能分为三层：

- P0：没有就无法成立 MVP
- P1：显著提升核心体验，应尽量纳入首版
- P2：有价值，但可在首版后迭代

## 6. P0 核心功能清单

### 6.1 连接管理

#### 功能项

- 新建 MySQL 连接
- 新建 PostgreSQL 连接
- 新建 SQLite 连接
- 编辑已保存连接
- 删除已保存连接
- 测试连接
- 连接失败错误提示

#### 价值说明

这是产品主流程的起点，没有稳定的连接能力，后续浏览与查看都无法成立。

#### 验收标准

- 用户能选择数据库类型并看到对应连接表单
- 必填参数有清晰说明
- 点击测试连接后能明确反馈成功或失败
- 失败时可展示基础原因，如主机不可达、认证失败、文件不存在
- 保存成功后，连接可在连接列表中再次打开

### 6.2 连接列表与入口主页

#### 功能项

- 展示已保存连接
- 点击进入连接
- 展示“新建连接”主入口
- 空状态引导

#### 价值说明

这是用户进入产品后的第一界面，决定产品是否“轻量且容易上手”。

#### 验收标准

- 首次打开时能看到清晰的新建连接引导
- 有连接数据时能快速识别连接名称和数据库类型
- 用户能从主页一键进入连接

### 6.3 数据库对象树浏览

#### 功能项

- 展示数据库 / Schema / 表 / 视图层级
- 展开与收起
- 刷新当前层级
- 加载状态反馈

#### 价值说明

对象树是“知道数据库里有什么”的核心入口，是产品浏览体验的中轴。

#### 验收标准

- 用户进入连接后能看到对象树
- 用户可展开层级并浏览对象
- 加载中的状态明确可感知
- 刷新后数据可重新拉取

### 6.4 表结构查看

#### 功能项

- 查看字段名称
- 查看字段类型
- 查看是否可为空
- 查看默认值
- 查看主键信息

#### 价值说明

用户判断“这张表是什么、字段怎么用”时，结构查看是最基础的需求。

#### 验收标准

- 用户选中表后可打开结构视图
- 字段列表可稳定展示
- 主要字段属性信息清晰、可扫描

### 6.5 表数据浏览

#### 功能项

- 打开表数据
- 默认分页加载
- 刷新数据
- 基础排序

#### 价值说明

这是 MVP 的核心能力之一，决定用户能否真正把它当成日常查看工具。

#### 验收标准

- 用户选中表后可直接查看数据
- 默认只加载有限结果，避免卡死
- 用户可以刷新数据
- 用户可以按单列做基础排序

### 6.6 基础查询

#### 功能项

- 提供简单 SQL 输入区域
- 执行单条查询语句
- 展示结果表格
- 展示执行错误信息

#### 价值说明

虽然产品不主打 SQL IDE，但没有基础查询能力，用户会在不少场景下被迫切换工具。

#### 验收标准

- 用户能进入查询区域并输入 SQL
- 单条 `SELECT` 查询可执行
- 查询结果能稳定展示为表格
- SQL 错误或执行错误有基础反馈

### 6.7 CSV 导出

#### 功能项

- 导出当前查询结果为 CSV
- 导出当前表格结果为 CSV

#### 价值说明

导出是轻度用户和个人开发者的常见需求，成本低但感知价值高。

#### 验收标准

- 用户可以从当前结果视图触发导出
- 导出成功后能在本地得到合法 CSV 文件
- 导出失败时有错误提示

## 7. P1 重要增强功能

### 7.1 对象搜索

#### 功能项

- 按对象名称搜索表或视图

#### 原因

当数据库对象较多时，搜索能显著缩短定位时间。

#### 建议

尽量纳入首版，如时间不足，可延后到首个小版本。

### 7.2 最近连接

#### 功能项

- 记录最近打开的连接

#### 原因

个人用户的数据库访问场景通常高度重复，这项能力能明显提升复用效率。

### 7.3 列宽调整

#### 功能项

- 在结果表格中调整列宽

#### 原因

数据查看可读性直接影响工具实用性，这是一项低复杂度高收益能力。

### 7.4 关键字筛选

#### 功能项

- 在当前结果集中做基础关键字过滤

#### 原因

对轻度用户来说，比写 SQL 更直观。

## 8. P2 后续迭代功能

以下功能不建议放进 MVP，但适合在产品验证后扩展：

- 多标签页浏览
- 查询历史
- 收藏常用表
- 保存常用 SQL
- 深色主题
- 连接分组
- 只读模式增强
- 更丰富的对象元信息展示

## 9. 明确不纳入 MVP 的功能

以下功能当前阶段明确不做：

- ER 图
- 数据建模
- 数据迁移管理
- 存储过程 / 函数管理
- 团队共享连接
- 权限审批
- 审计日志体系
- 插件市场
- 可视化 SQL 构建器
- 大规模多数据库兼容平台

## 10. MVP 页面级清单

### 10.1 连接主页

必须包含：

- 新建连接入口
- 已保存连接列表
- 空状态提示

### 10.2 新建 / 编辑连接页

必须包含：

- 数据库类型选择
- 不同数据库的连接参数表单
- 测试连接按钮
- 保存按钮
- 错误提示区

### 10.3 数据库浏览主界面

必须包含：

- 左侧对象树
- 中央主内容区
- 基础刷新入口

### 10.4 表结构视图

必须包含：

- 字段列表
- 字段属性信息

### 10.5 表数据视图

必须包含：

- 数据表格
- 分页或加载控制
- 排序
- 刷新
- 导出入口

### 10.6 查询视图

必须包含：

- SQL 输入区
- 执行按钮
- 结果区
- 错误信息区
- 导出入口

## 11. MVP 技术支撑清单

虽然以下能力不一定直接被用户感知，但属于 MVP 必备支撑：

### 11.1 本地连接配置存储

- 能持久化保存连接信息
- 能在重启应用后恢复连接列表

### 11.2 数据库驱动适配层

- 为 MySQL / PostgreSQL / SQLite 提供统一接入方式
- 对元数据获取和结果集读取做基础封装

### 11.3 异常处理与状态管理

- 连接失败、查询失败、超时等错误必须可反馈
- 加载状态必须清晰可见

### 11.4 大结果保护

- 默认分页或限制返回条数
- 避免一次性读取过大结果导致卡顿

## 12. 推荐开发顺序

为了尽快做出可验证版本，建议按以下顺序推进：

1. 连接管理
2. 连接主页与连接列表
3. 对象树浏览
4. 表结构查看
5. 表数据浏览
6. 基础查询
7. CSV 导出
8. 对象搜索与最近连接

这个顺序能保证每完成一段，就离完整闭环更近一步。

## 13. MVP 发布门槛

只有在满足以下条件时，才建议视为可发布 MVP：

- 三类数据库至少完成基础连接与浏览闭环
- 表结构查看稳定可用
- 表数据分页查看稳定可用
- 基础查询稳定可用
- 导出能力可用
- 关键错误都有明确反馈
- 核心流程无明显阻塞性缺陷

## 14. 结论

Open DB Viewer 的 MVP 应该坚持“小而完整”而不是“大而分散”。

最重要的不是功能数量，而是让用户真正感受到：

- 打开就能连
- 连上就能看
- 看完就能查

只要把这条主链路做顺，产品就具备进一步演进为优秀数据库浏览工具的基础。
