## 2. 背景与问题定义

当前 Open DB Viewer 已完成 MVP 主链路，但界面仍停留在“功能拼接”阶段，主要问题不是单个控件不够美观，而是整体缺少统一的产品界面框架。

当前 UI 的核心问题包括：

- 页面像多个功能块堆叠，缺少稳定的工作台感
- 信息层级弱，用户不容易快速理解“我当前在看什么、接下来能做什么”
- 高优先级主路径没有被强化，连接、浏览、查看、查询都处于同等视觉权重
- 视觉语言不统一，按钮、状态区、内容区之间缺少清晰层次
- 空态、默认态、加载态虽然已有提示，但仍不像一个成熟产品的状态系统

因此，这次 UI 补齐不应理解为“换皮”，而应理解为一次以产品体验为目标的界面重构。

## 3. UI 重构目标

本次 UI 产品计划重点解决以下 5 个目标：

### 3.1 建立稳定的工作台框架

用户打开应用后，不再进入一个零散页面，而是进入一个结构稳定的桌面工作台。不同任务切换时，界面骨架保持稳定，降低认知切换成本。

### 3.2 强化主路径

产品要清晰服务于以下主路径：

1. 打开应用
2. 选择或新建连接
3. 浏览对象树
4. 选择目标表
5. 查看结构 / 数据
6. 执行查询
7. 导出结果

所有页面结构、视觉强调和操作区布置都应优先服务这条路径。

### 3.3 建立极简效率型视觉语言

产品整体视觉调性确定为：

- 极简效率型
- 工作台首页
- 宽松舒展
- 静音工作台风格

视觉不追求花哨，也不追求 IDE 式的专业压迫感，而要让用户觉得：

- 打开就能用
- 界面安静
- 重点清楚
- 长时间看不累

### 3.4 统一页面状态体验

让空态、加载态、成功态、错误态成为产品体验的一部分，而非开发补丁。尤其是数据库工具的默认态和未选中态，需要具备足够的完成度。

### 3.5 为后续 UI 扩展建立基础

后续如果加入：

- 多标签页
- 收藏对象
- 查询历史
- 深色主题

本次 UI 方案也应具备向这些能力扩展的结构基础，而不是每次继续拼接。

## 4. 竞品研究摘要

本节不做完整视觉史分析，只聚焦对 Open DB Viewer 有指导意义的结论。

### 4.1 Navicat

可借鉴点：

- 上手门槛低
- 主操作清楚
- 管理型任务的可见性较好

不建议直接借鉴的点：

- 传统桌面工具感较强
- 视觉语言偏旧
- 信息组织容易偏碎片化

对本产品的启发：

- “易上手”比“显得专业”更重要
- 主按钮和常用动作要容易找到
- 数据库管理能力不必重，但高频入口要非常清楚

### 4.2 DBeaver

可借鉴点：

- 对象树层级成熟
- 数据库浏览逻辑完整
- 多数据库视角下的组织能力清晰

不建议直接借鉴的点：

- 首屏复杂度高
- 信息密度较大
- 容易让轻度用户有畏难感

对本产品的启发：

- 左侧对象导航必须专业、稳定、可扩展
- 但不能把整个产品做成“数据库平台”

### 4.3 DataGrip

可借鉴点：

- 界面秩序感强
- 工作区边界清楚
- 专业感来自结构，而不是花哨装饰

不建议直接借鉴的点：

- IDE 气质明显
- 对 SQL 重度用户更友好
- 首次使用学习成本偏高

对本产品的启发：

- 可以学习其秩序和布局，不学习其“开发环境化”
- 工作区必须稳定、清楚、有上下文

### 4.4 竞品结论

Open DB Viewer 的 UI 方向不应该是上述竞品的平均值，而应该是：

- 用 Navicat 的低门槛
- 借 DataGrip 的秩序感
- 吸收 DBeaver 的对象树组织方式

最终形成：

“一个更轻、更静、更容易上手的数据库工作台”

## 5. UI 总策略

本产品 UI 总策略定义为：

## 静音工作台

其核心含义如下：

### 5.1 工作台优先于页面

打开应用后，用户应该感受到自己进入了一个稳定的工作环境，而不是在多个临时页面之间跳转。

### 5.2 克制优先于装饰

主要依靠以下手段建立品质感：

- 间距
- 对齐
- 文字层级
- 背景层级
- 选中态
- 状态反馈

而不是依赖大面积颜色、复杂边框或装饰图形。

### 5.3 安静优先于刺激

数据库浏览属于中高频、长时任务。界面应减少视觉噪音，让用户更专注于对象树、表结构和数据结果，而不是被 UI 本身干扰。

### 5.4 默认态也要高级

产品中大量时间会停留在以下状态：

- 还没有连接
- 当前未选表
- 当前没有查询结果
- 当前数据为空

因此默认态不是补充页，而是主体验之一。

### 5.5 主路径永远最短

用户完成“连库、找表、看数据”的路径应尽量保持短而直。

## 6. 视觉语言规范

### 6.1 整体视觉关键词

- 极简
- 安静
- 有秩序
- 宽松
- 柔和但不幼稚
- 专业但不沉重

### 6.2 色彩策略

建议采用中性色主导、单强调色辅助的体系。

推荐方向：

- 页面背景：轻微偏冷的浅灰底
- 主面板背景：更高亮度的浅色卡片
- 一级文字：深灰而非纯黑
- 二级文字：中灰
- 分隔线：低对比浅灰
- 强调色：蓝绿 / 墨青 / 清透蓝，不使用紫色
- 成功 / 错误 / 警示色：只在状态中使用，不参与常规布局竞争

原则：

- 强调色只服务“当前选中”“主操作”“焦点状态”
- 不允许页面出现过多彩色按钮竞争注意力

### 6.3 字体与排版

建议避免明显的系统默认感，优先使用有现代效率工具气质的字体方案。

建议方向：

- 中文：优先选择清晰、现代、笔画稳定的无衬线字体
- 英文 / 数字 / SQL：与中文搭配协调的现代无衬线或等宽字体

层级建议：

- 页面标题：明确，但不夸张
- 模块标题：比正文略强
- 表格标题：强调可扫描性
- 辅助说明：轻但清楚

目标不是“设计感字体秀”，而是增强产品完成度。

### 6.4 圆角、边框、阴影

建议：

- 圆角：中等偏大，增强现代效率工具气质
- 边框：少量使用，更多依赖背景层级区分模块
- 阴影：非常轻，避免卡片漂浮感过重

### 6.5 间距体系

这是本次 UI 补齐最关键的部分之一。

建议建立稳定的间距节奏：

- 页面外边距
- 模块间距
- 表单分组间距
- 列表项间距
- 内容区上下留白

视觉的“宽松舒展”主要依赖间距与模块节奏，而不是简单放大空白。

## 7. 全局框架设计

### 7.1 应用壳层结构

建议统一为如下结构：

```text
+----------------------------------------------------------------------------------+
| 顶部栏：产品名 | 当前位置 | 主操作区 | 当前连接信息                               |
+------------------------------+---------------------------------------------------+
| 左侧边栏                     | 中央主工作区                                    |
| 连接 / 对象导航              | 当前页面标题 / 面包屑 / tab / 内容              |
| 搜索 / 快捷动作              |                                                   |
|                              |                                                   |
+------------------------------+---------------------------------------------------+
| 底部状态条：加载状态 / 查询状态 / 导出状态 / 错误提示                            |
+----------------------------------------------------------------------------------+
```

### 7.2 顶部栏

顶部栏不做传统菜单栏替代，而做“当前上下文 + 主操作”区。

建议包含：

- 产品名或简洁标识
- 当前视图标题
- 当前连接信息
- 主操作按钮
  - 新建连接
  - 刷新
  - 导出
- 预留设置入口

### 7.3 左侧边栏

左侧边栏是整个产品中最重要的结构区之一。

其职责不是“放树”，而是承接所有入口信息：

- 已保存连接
- 当前活动连接
- 数据对象树
- 搜索
- 快捷操作

左侧边栏应该长期存在，不随页面频繁跳出跳进。

### 7.4 中央主工作区

中央区域只专注一件事：

- 首页时承接工作台内容
- 浏览时承接结构 / 数据 / 查询
- 新建连接时承接连接配置面板

不应将多个同级内容在纵向无限堆叠。

### 7.5 底部状态条

底部状态区应升级为统一反馈带，而不是一段普通文字。

建议承接：

- 当前动作
- 查询耗时
- 导出结果
- 错误恢复提示

## 8. 核心页面 UI 方案

### 8.1 工作台首页

#### 页面目标

- 首次用户能立刻开始
- 老用户能直接进入连接
- 首页本身就具备“长期工作台”的感觉

#### 页面结构

- 左栏：连接列表、最近连接、快速新建
- 主区：
  - 欢迎标题
  - 产品一句话说明
  - 主操作卡片
  - 最近连接快捷入口
  - 空态提示或推荐动作

#### 设计重点

- 不做“营销首页”，而做“可进入工作的首页”
- 主按钮清晰，但不突兀
- 空态时也保持完成度

### 8.2 新建连接面板

#### 页面目标

把“配置数据库连接”从一套技术表单，变成一个低压力、低错误率的任务面板。

#### 页面结构

- 顶部：返回 / 标题 / 说明
- 中段：
  - 数据库类型切换
  - 基础信息组
  - 连接信息组
- 底部：
  - 测试连接
  - 保存连接
  - 状态反馈

#### 设计重点

- 数据库类型切换做成 segmented control
- SQLite 和 MySQL / PostgreSQL 表单分开组织
- “测试连接成功”后应显著降低保存阻力
- 底部操作区建议固定，减少滚动造成的操作断裂

### 8.3 数据库浏览主界面

#### 页面目标

成为产品最核心的工作台场景，让浏览对象和查看内容的关系一眼成立。

#### 页面结构

- 左侧：对象树
- 中间顶部：标题、面包屑、当前表信息
- 中间主体：tab 工作区

建议使用如下 tab：

- 结构
- 数据
- 查询

#### 设计重点

- 不要把结构、数据、查询全部纵向堆叠
- 选中表以后，应明确告诉用户自己当前在看哪张表
- 结构 / 数据 / 查询之间应是平级工作模式切换

### 8.4 表结构视图

#### 页面目标

让用户快速理解表字段构成，并具备良好可扫描性。

#### 页面结构

- 顶部：表名、所属连接 / 库信息、辅助说明
- 主体：字段表格

#### 设计重点

- 字段表格重扫描性，不做重装饰
- 主键、类型、是否可空等关键列要易识别
- 表为空或无元信息时要给出温和空态

### 8.5 表数据视图

#### 页面目标

让数据浏览保持轻松、可控、有秩序。

#### 页面结构

- 顶部工具栏：
  - 刷新
  - 导出
  - 分页信息
  - 可选筛选入口
- 主体：数据表格
- 底部：分页区或结果信息

#### 设计重点

- 表格应减少视觉硬边框，提升呼吸感
- 标题行、悬浮行、选中行要区分明确
- 空数据时提供明确反馈，而不是显示空白表格

### 8.6 查询工作区

#### 页面目标

提供轻量但可信赖的 SQL 工作区，不走 IDE 路线。

#### 页面结构

- 顶部操作条：执行 / 清空 / 导出
- 中部：SQL 输入面板
- 下部：结果区
- 辅助：状态反馈条

#### 设计重点

- 输入区更像高级命令面板，而不是代码 IDE
- 查询结果与输入区关系清楚
- 错误态与无结果态要简洁、有恢复路径

## 10. 状态系统设计

### 10.1 Loading

适用场景：

- 连接列表加载
- 测试连接
- 对象树加载
- 表结构加载
- 表数据加载
- SQL 执行

建议：

- 页面级 loading 采用骨架或轻提示
- 按钮级 loading 保留在关键动作上
- 避免全屏强打断式 loading

### 10.2 Empty

适用场景：

- 没有保存连接
- 没有活动连接
- 未选表
- 表无数据
- 查询无结果

建议：

- 空态必须包含“当前状态解释”
- 尽量附带下一步动作
- 空态文案用自然语言，不用底层术语

### 10.3 Error

适用场景：

- 连接失败
- 元数据读取失败
- SQL 执行失败
- 导出失败

建议：

- 错误信息要可理解
- 优先提供“重试 / 返回 / 重新编辑”的恢复动作
- 不让错误面板挤压主内容到失控

### 10.4 Success

适用场景：

- 连接测试成功
- 保存成功
- 导出成功

建议：

- 成功反馈应轻量，不打断操作
- 优先用内联状态或底部反馈，不用频繁弹窗

## 11. 交互与动效原则

### 11.1 微动效即可

只保留少量有意义的动效：

- 页面区域淡入
- 卡片轻微显现
- 选中态平滑切换
- hover 状态柔和过渡

### 11.2 不做炫技动画

数据库工具的动效目标是降低僵硬感，不是制造存在感。

### 11.3 优先反馈而不是表演

所有动效都应该帮助用户判断：

- 我点到了哪里
- 当前系统在干什么
- 哪个对象正在被选中

## 12. 实施优先级与阶段计划

建议按 4 个阶段推进 UI 补齐。

### Phase 1：建立视觉基线

目标：

- 定义视觉语言
- 建立设计令牌
- 明确全局框架

产出：

- 颜色、排版、间距、状态规范
- 全局布局草图
- UI 重构原则说明

### Phase 2：重构产品壳层

目标：

- 先解决“产品不像产品”的问题

产出：

- 新 `AppShell`
- 顶部栏
- 左侧导航区
- 底部状态条
- 首页工作台结构

### Phase 3：重做核心主路径页面

目标：

- 让“连接 -> 浏览 -> 查看 -> 查询”这条链路顺起来

产出：

- 新建连接面板
- 浏览页 tab 工作区
- 表结构视图优化
- 表数据视图优化
- 查询工作区优化

### Phase 4：细节打磨与状态完善

目标：

- 从“骨架正确”升级到“愿意常用”

产出：

- 空态体系
- 动效微调
- 文案与反馈细化
- 细节一致性修正

## 13. 对研发的直接要求

为了让 UI 重构真正落地，研发实现时应遵循以下约束：

- 先建立全局样式变量，不要直接散写页面样式
- 先做布局壳层和复用组件，不要逐页临时堆样式
- 浏览页必须改成 tab 化内容工作区
- 首页必须改成工作台首页，而不是简单按钮页
- 查询区不要做成 IDE 风格
- 颜色和视觉强调必须克制，避免“补一处亮一处”

## 14. 成功标准

完成本次 UI 补齐后，应达到以下体验结果：

- 用户打开应用后，能立即理解它是一个数据库工作台
- 首次用户知道从哪里开始
- 老用户可以直接进入高频任务
- 浏览页层级明显，结构 / 数据 / 查询不再互相抢占
- 空态、加载态、成功态、错误态具备一致性
- 产品气质从“功能 demo”升级为“轻量效率工具”

## 15. 结论

Open DB Viewer 的 UI 补齐，不应走“重型数据库 IDE”路线，也不应停留在“极简但简陋”的路线。

最合适的方向是：

- 极简效率型
- 工作台首页
- 宽松舒展
- 静音工作台
