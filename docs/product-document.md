# Open DB Viewer 产品文档

## 1. 文档信息

- 产品名称：Open DB Viewer
- 产品阶段：0 到 1
- 文档类型：产品文档（PRD / 产品定义）
- 目标版本：V1.0
- 文档日期：2026-04-15

## 2. 产品概述

Open DB Viewer 是一款面向个人用户的轻量数据库浏览软件，聚焦于 MySQL、PostgreSQL 和 SQLite 三类主流关系型数据库，帮助用户以更低的学习成本完成数据库连接、库表浏览、结构查看和数据查询。

产品不以“全能数据库 IDE”为目标，也不在第一阶段追求复杂的数据库开发、建模、协作与运维能力，而是优先解决用户在日常工作中最常见的问题：

- 快速连上数据库
- 快速知道数据库里有什么
- 快速查看某张表的数据

一句话定位：

“一个面向个人开发者与轻度数据使用者的轻量数据库浏览工具，帮助用户以最低学习成本快速连接 MySQL、PostgreSQL 和 SQLite，完成库表浏览、结构查看与数据查询。”

## 3. 产品定位

### 3.1 定位陈述

Open DB Viewer 是一个“轻量、易上手、聚焦浏览”的桌面数据库工具。

它的核心价值不在于提供最强大的 SQL IDE 能力，而在于让用户在打开软件后尽快完成以下主流程：

1. 新建或打开一个数据库连接
2. 浏览数据库对象
3. 查看表结构
4. 查看表数据
5. 进行简单筛选或基础查询

### 3.2 产品关键词

- 轻量
- 易上手
- 快速连接
- 清晰浏览
- 低认知负担
- 面向个人效率

### 3.3 产品边界

第一版强调“查看”和“轻查询”，不强调“深度开发”和“团队协作”。

产品不是：

- DataGrip 式的专业数据库 IDE
- DBeaver 式的大而全数据库平台
- Navicat 式覆盖大量企业功能的数据库管理套件

产品更接近：

- 一个更完整、更稳定的个人数据库浏览器
- 一个比命令行更直观、比重型工具更轻便的日常数据库助手

## 4. 目标用户

### 4.1 核心用户画像

#### 用户类型 A：个人开发者

特征：

- 经常连接本地开发库、测试库或云数据库
- 需要查看表结构、检查数据、排查问题
- 希望工具启动快、配置简单、操作直观

核心诉求：

- 快速建连
- 快速看表
- 快速查数据

#### 用户类型 B：独立开发者

特征：

- 通常一个人维护产品、后端和数据
- 对工具的“效率 / 成本 / 学习门槛”比较敏感
- 不希望为偶发数据库查看场景付出太高工具复杂度

核心诉求：

- 少配置
- 少记忆
- 少切换工具

#### 用户类型 C：轻度数据使用者

特征：

- 可能不是专业 DBA 或后端工程师
- 需要偶尔查看表字段、验证数据、导出部分结果
- 对 SQL 和数据库概念了解有限

核心诉求：

- 界面清晰
- 操作有引导
- 不容易误操作

### 4.2 非目标用户

以下用户不作为第一版重点服务对象：

- 需要复杂 SQL 开发和深度调优的资深数据库开发者
- 需要数据库建模、迁移管理、版本发布的团队
- 需要审批、审计、权限流转的企业客户
- 需要同时管理大量异构数据库和大规模连接的 DBA 团队

## 5. 用户痛点

当前用户在使用现有数据库工具时，常见问题包括：

### 5.1 工具过重

- 启动慢
- 功能过多导致入口复杂
- 首次使用需要理解大量概念和配置

### 5.2 主流程不够短

- 建立连接步骤多
- 找到目标表路径深
- 看结构和看数据之间切换繁琐

### 5.3 信息呈现不够友好

- 树结构层级复杂
- 字段信息阅读成本高
- 结果表格可读性差

### 5.4 轻度用户有畏难感

- 工具术语偏专业
- SQL 编辑能力太重，反而增加心理负担
- 不知道下一步该点哪里

## 6. 核心价值主张

Open DB Viewer 首版要建立的核心价值如下：

### 6.1 更快连接数据库

- 用更短的配置流程完成连接创建
- 提供明确的连接测试反馈
- 让常见连接参数更容易理解

### 6.2 更快理解数据库结构

- 用清晰的数据库树展示库、Schema、表、视图
- 用简洁的结构面板展示字段、类型、主键、索引等基础信息
- 降低用户从“连上库”到“找到目标表”的时间成本

### 6.3 更快查看数据

- 一键打开表数据
- 默认分页加载
- 提供轻量筛选、排序和刷新能力

### 6.4 更低学习成本

- 界面语言尽量贴近日常理解
- 核心功能入口前置
- 减少专业能力对主流程的干扰

## 7. 竞品参考与产品策略

本产品参考 Navicat、DBeaver 和 DataGrip，但不复制其能力全集。

### 7.1 Navicat

可借鉴点：

- 上手门槛低
- 常用操作可视化程度较高
- 更贴近日常管理使用习惯

不优先复制的点：

- 面向多场景的大量管理功能
- 偏企业化的完整管理工具链

### 7.2 DBeaver

可借鉴点：

- 数据库对象结构完整
- 浏览树组织成熟
- 多数据库支持思路清晰

不优先复制的点：

- 产品功能面广，界面复杂度相对较高
- 首次上手的信息密度偏大

### 7.3 DataGrip

可借鉴点：

- 专业感强
- SQL 体验成熟
- 数据对象理解能力强

不优先复制的点：

- IDE 化明显
- 学习成本偏高
- 更适合重度数据库开发者

### 7.4 产品策略结论

Open DB Viewer 的差异化策略是：

- 不做功能最多
- 不做 SQL 最强
- 优先做“连接最顺、浏览最清楚、查看最轻松”

差异化表达可总结为：

“比重型工具更轻、更快上手；比简陋工具更完整、更稳定。”

## 8. 使用场景

### 8.1 场景一：开发调试时快速查看表数据

用户在本地开发或联调时，需要确认某张表是否写入成功、字段值是否正确。

目标：

- 迅速连接数据库
- 找到目标表
- 查看最近数据或筛选记录

### 8.2 场景二：排查线上或测试环境问题

用户需要用只读方式连接测试库或线上库，快速确认数据状态。

目标：

- 稳定连接
- 快速定位表
- 避免误修改

### 8.3 场景三：理解陌生数据库结构

用户接手一个新项目或第三方数据库，需要先搞清楚有哪些表、字段如何组织。

目标：

- 清楚浏览数据库树
- 快速查看表结构
- 理解字段含义和类型

### 8.4 场景四：轻度用户查看数据结果

产品、运营、分析等轻度用户需要查看一张表或执行简单查询，验证业务数据。

目标：

- 不依赖复杂 SQL 知识
- 尽量通过界面操作完成查看
- 导出少量结果用于沟通

## 9. 首版产品目标

### 9.1 业务目标

- 建立产品的差异化定位
- 形成一个可被个人开发者接受的基础数据库浏览工具
- 为后续扩展 SQL 工作台和更高阶能力建立用户基础

### 9.2 用户目标

用户应能在第一版中高效完成：

- 新建连接
- 测试连接
- 浏览数据库对象
- 查看表结构
- 查看表数据
- 做简单查询和结果导出

### 9.3 体验目标

- 首次使用不需要学习复杂概念
- 核心路径清晰，尽量 1 到 3 次点击到达目标
- 重要界面简洁，不让低频能力干扰高频任务

## 10. 功能范围

## 11. 产品信息架构

建议首版采用三栏式或双栏增强布局。

### 11.1 一级结构

- 左侧：连接与数据库对象树
- 中央：主工作区
- 右侧或底部：对象详情 / 结果状态 / 错误信息

### 11.2 主导航结构

一级视图建议包括：

- 连接主页
- 数据库浏览页
- 表结构页
- 表数据页
- 查询页

### 11.3 核心对象层级

建议基础层级如下：

- 连接
- 数据库

- 字段 / 索引

## 12. 核心用户流程

### 12.1 流程一：首次连接数据库

1. 用户打开应用
2. 点击“新建连接”
3. 选择数据库类型
4. 填写必要连接信息
5. 点击“测试连接”
6. 连接成功后保存
7. 进入数据库浏览界面

成功标准：

- 用户能理解需要填写什么
- 连接错误能被清晰提示
- 成功后自动进入下一步浏览

### 12.2 流程二：查看某张表的数据

1. 用户在左侧展开连接
2. 浏览数据库 / Schema / 表
3. 选择目标表
4. 默认打开“数据”或“结构”页
5. 查看前 N 条数据
6. 根据需要筛选、排序或刷新

成功标准：

- 用户能快速找到目标表
- 数据加载反馈明确
- 表格内容具备基础可读性

### 12.3 流程三：查看表结构

1. 选择目标表
2. 打开结构页签
3. 查看字段、类型、主键、默认值等信息

成功标准：

- 结构信息完整且清晰
- 字段信息具备可扫描性

### 12.4 流程四：执行基础查询

1. 打开查询页
2. 输入一条简单查询
3. 点击执行
4. 查看结果或错误提示
5. 需要时导出结果

成功标准：

- 执行入口清晰
- 错误反馈易懂
- 结果展示稳定

## 13. 核心交互原则

### 13.1 先展示高频能力

高频动作如“新建连接”“刷新”“查看数据”“查看结构”应放在视觉和交互前列。

### 13.2 避免过度专业化术语

尽量用用户更容易理解的表达，如：

- 新建连接
- 测试连接
- 查看结构
- 查看数据

而不是过多暴露偏底层的数据库术语。

### 13.3 默认安全

第一版偏向“浏览型工具”，应尽量避免默认提供高风险修改能力，减少误操作概率。

### 13.4 状态反馈清晰

在连接、加载、执行查询、导出等关键操作中，需要明确反馈：

- 正在进行中
- 成功
- 失败
- 失败原因

## 14. 非功能需求

### 14.1 性能

- 应用启动速度应明显快于重型 IDE 工具
- 常规数据库树展开和表数据查看响应应保持流畅
- 表数据应采用分页或懒加载，避免一次性加载过多数据

### 14.2 易用性

- 首次用户应能在无教程情况下完成基本连接与查看
- 关键操作路径尽量缩短
- 错误提示尽量具备可操作性

### 14.3 稳定性

- 连接失败、超时、权限不足等情况需要可识别、可恢复
- 查询失败不能导致应用界面卡死
- 多次切换连接和对象时应保持稳定

### 14.4 安全性

- 连接信息应本地安全存储
- 密码字段应做基础保护
- 默认鼓励使用只读连接查看线上数据

## 16. 风险与取舍

### 16.1 风险一：功能贪多

如果过早追求和 Navicat / DBeaver / DataGrip 对齐，会导致首版失焦。

应对原则：

- 坚持只服务核心主流程
- 非高频需求进入后续版本

### 16.2 风险二：轻量但能力过弱

如果只强调轻量，可能变成“只能看、但不好用”的工具。

应对原则：

- 在轻量的前提下，保证连接、浏览、数据查看三个能力足够完整
- 首版至少提供基础查询和结果导出

### 16.3 风险三：多数据库差异带来复杂度

MySQL、PostgreSQL、SQLite 的对象结构和元信息展示存在差异。

应对原则：

- 首版统一主流程
- 差异能力延后处理
- 优先保证通用浏览体验一致性

## 17. 成功标准

产品首版是否成功，可先用以下标准衡量：

- 用户能在短时间内完成连接创建
- 用户能快速找到目标表并看到结构与数据
- 用户不需要阅读复杂文档即可完成高频任务
- 用户愿意把它作为本地或测试环境的常用数据库查看工具

## 18. 总结

Open DB Viewer 的首版方向不是做“更大的数据库工具”，而是做“更顺手的数据库浏览工具”。

它的核心竞争力应来自三个方面：

- 连接快
- 浏览清楚
- 上手轻松

只要第一版能把“快速连接数据库、快速看清库表、快速查看数据”这一主链路打磨好，就能建立明确的产品价值，并为后续扩展查询、效率增强和安全能力打下基础。
